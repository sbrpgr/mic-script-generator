# 에이전트 작동 명세

이 문서는 코워크스페이스 작업을 맡은 에이전트가 기능 추가, 수정, 삭제, 배포, 보안 점검을 빠뜨리지 않도록 따르는 운영 기준이다. 목표는 빠른 처리보다 플랫폼의 안정성, 완성도, 개인정보 보호, 반복 가능한 검증을 우선하는 것이다.

## 기본 원칙

- 코워크스페이스는 정적 사이트와 브라우저 내부 처리 중심으로 운영한다.
- 사용자가 입력한 텍스트, 파일, 이미지, PDF, 자막, 음성, 영상 데이터는 자체 서버로 보내지 않는 것을 기본값으로 한다.
- 새 서버, DB, 외부 API, 유료 서비스, 계정 연동이 필요하면 먼저 필요성과 대안을 검토하고 사용자에게 명확히 알린다.
- 작업 중 사용자가 열어둔 별도 자료나 미추적 파일은 요청이 없으면 건드리지 않는다.
- 기존 사용자 변경을 되돌리지 않는다.
- 기능보다 신뢰가 먼저다. 모호한 경우 개인정보, 보안, UX 안정성 쪽으로 보수적으로 판단한다.

## 작업 시작 루틴

1. `git status --short`로 작업 트리를 확인한다.
2. 관련 파일과 기존 패턴을 먼저 읽는다.
3. 변경 범위를 좁게 잡고, 영향을 받는 문서와 테스트를 함께 확인한다.
4. 기능 변경이면 캐시 버전, sitemap, 구조화 데이터, 카테고리 페이지, 분석 태그, 광고 슬롯, 보안 헤더 영향까지 함께 본다.
5. 사용자가 명시하지 않은 대규모 리팩터링은 하지 않는다.

## 기능 추가 체크리스트

새 도구를 추가할 때는 다음 항목을 빠뜨리지 않는다.

- `app.js`의 `TOOL_DEFS`
- `TOOL_RENDERERS`
- `TOOL_VISUALS`
- `TOOL_USE_EXAMPLES`
- `TOOL_EXTRA_FAQS`
- `tools/{slug}/index.html`
- `sitemap.xml`
- 관련 카테고리 랜딩 페이지 설명
- 사용 흐름과 오류 안내 문구
- 파일 업로드 도구라면 파일 선택, 드래그앤드롭, 붙여넣기 가능 여부
- 결과 다운로드 파일명 정리
- `dayway.ict@gmail.com` 연락 안내가 필요한 실패 상태
- 브라우저 처리 원칙과 개인정보 영향
- 새 라이브러리 사용 시 CSP와 의존성/보안 영향

## 기능 수정 체크리스트

- 문제 재현 조건을 먼저 확인한다.
- 같은 패턴을 쓰는 다른 도구에 같은 문제가 있는지 검색한다.
- 공통 유틸로 고치는 것이 더 안전하면 공통화하되, 불필요한 구조 변경은 피한다.
- 기존 UX 문구와 스타일을 맞춘다.
- 회귀 가능성이 있으면 `scripts/smoke-test.js` 또는 `scripts/project-audit.js`에 검사를 추가한다.

## 기능 삭제 또는 경로 변경 체크리스트

- `TOOL_DEFS`, 렌더러, 비주얼, 예시, FAQ에서 제거한다.
- `tools/{slug}/index.html`과 sitemap을 정리한다.
- 기존 URL이 검색엔진에 노출된 경우 `_redirects` 또는 대체 안내 필요성을 검토한다.
- 블로그 홍보 자료, README, ROADMAP, 관련 문서의 링크를 확인한다.
- Search Console 색인 요청 또는 제거가 필요한지 기록한다.

## 업로드 UX 기준

- 파일 기반 도구는 가능한 한 파일 선택과 드래그앤드롭을 모두 지원한다.
- 스크린샷 기반 이미지 도구는 붙여넣기 가능 여부를 우선 검토한다.
- `accept` 조건과 실제 처리 가능한 형식이 일치해야 한다.
- 지원하지 않는 파일은 조용히 실패하지 않고 명확히 안내한다.
- 브라우저가 드래그 파일 주입을 지원하지 않는 경우 파일 선택 버튼을 안내한다.

## 분석, 광고, SEO 기준

- Google 태그는 `scripts/site-tags.js`로 관리한다.
- 사용자 입력값, 파일명, 추출 결과, QR 결과, 녹화 내용 등 민감할 수 있는 값은 분석 이벤트로 보내지 않는다.
- 광고는 작업 영역, 업로드 박스, 텍스트 입력 영역, 결과 미리보기 안에 넣지 않는다.
- 새 HTML 페이지는 canonical, meta description, 구조화 데이터, sitemap을 확인한다.
- `app.js` 또는 `styles.css`가 바뀌면 모든 HTML과 `site.webmanifest`의 캐시 버전을 올린다.

## 보안 기준

- 클라이언트 코드에 토큰, API 키, 비밀값을 넣지 않는다.
- `eval`, `Function`, `document.write`를 사용하지 않는다.
- 외부 링크 열기는 `noopener,noreferrer`를 사용한다.
- 사용자 입력을 HTML로 렌더링할 때는 반드시 escape 또는 안전한 렌더링 경로를 사용한다.
- Object URL은 사용 후 revoke한다.
- 외부 라이브러리는 필요한 경우에만 쓰고, CSP 허용 범위를 최소화한다.
- CSP에서 `script-src 'unsafe-inline'`을 추가하지 않는다.
- 권한이 필요한 기능은 권한 목적, 차단 상태, 실패 상태를 사용자에게 분명히 안내한다.

## 필수 검증

작업 완료 전 기본 검증은 다음을 기준으로 한다.

```bash
npm run check
git diff --check
```

`npm run check`는 다음을 포함해야 한다.

- JavaScript 문법 검사
- 사이트 태그 검사
- 프로젝트 구조 감사
- 스모크 테스트
- 의존성 취약점 감사

필요 시 추가로 확인한다.

- 브라우저 UI 흐름
- 모바일/데스크톱 레이아웃
- 실제 파일 처리
- 카메라/마이크 권한 기능의 실기기 테스트
- 운영 URL 200 응답과 새 캐시 버전 반영
- GitHub Actions 배포 성공

## 배포 기준

- 로컬 검증이 통과한 뒤 커밋한다.
- 커밋 메시지는 변경 목적을 짧고 명확하게 적는다.
- `origin/main`에 푸시한 뒤 GitHub Actions 배포 결과를 확인한다.
- 운영 사이트에서 새 캐시 버전 또는 주요 변경이 반영됐는지 확인한다.
- 배포 후 보고에는 변경 내용, 검증 결과, 남은 리스크를 짧게 남긴다.

## 보고 기준

사용자에게는 장황한 내부 로그보다 결과 중심으로 보고한다.

- 무엇을 고쳤는지
- 어떤 검증을 통과했는지
- 배포 여부
- 남은 수동 확인 또는 리스크
- 건드리지 않은 사용자 파일이 있으면 명시

## 중단 기준

다음 경우에는 임의로 진행하지 않고 사용자에게 알려야 한다.

- 서버, DB, 결제, 계정, 외부 API가 필요해지는 경우
- 개인정보 처리 방식이 달라지는 경우
- 광고 정책 또는 법적 문구 변경이 필요한 경우
- 기존 사용자 작업을 덮어쓸 위험이 있는 경우
- 자동 검증으로 확인할 수 없는 실기기/권한 문제가 핵심인 경우
